# 02. Chunking Strategies for VTT Files — Implementation Plan

## Overview

Implement a hybrid VTT chunking pipeline that parses `.vtt` subtitle files from class recordings, merges micro-cues into sentences, detects topic boundaries via keyword signals and silence gaps, builds semantically coherent chunks with overlap, and enriches each chunk with metadata and auto-tagged types.

---

## Architecture

```
.vtt file
   ↓
Phase 1: Parse & Clean
   ↓
Phase 2: Merge Micro-Cues
   ↓
Phase 3: Detect Topic Boundaries
   ↓
Phase 4: Build Chunks (with overlap)
   ↓
Phase 5: Enrich with Metadata
   ↓
Phase 6: Output & Validate
   ↓
chunks.json + chunks_summary.json
```

---

## File Structure

All chunking logic lives under `src/ingestion/`:

```
src/ingestion/
├── __init__.py
├── vtt_parser.py          ← Phase 1: parse & clean VTT into cue objects
├── cue_merger.py          ← Phase 2: merge short cues into sentence-level cues
├── topic_detector.py      ← Phase 3: detect topic boundaries via keywords + silence
├── chunk_builder.py       ← Phase 4: build chunks with overlap
├── metadata_enricher.py   ← Phase 5: add metadata + auto-tag chunk_type
└── pipeline.py            ← Orchestrates all phases, runs validation (Phase 6)
```

Config constants exposed in a single place:

```
src/ingestion/config.py    ← All tunable constants
```

Output goes to `data/chunks/`:

```
data/chunks/
├── chunks.json            ← Full chunk objects
└── chunks_summary.json    ← Summary subset
```

## Config Constants

| Constant | Default | Used In |
|---|---|---|
| `MERGE_GAP_THRESHOLD_SECONDS` | 3 | Phase 2 — max gap to merge consecutive cues |
| `MERGE_MAX_CHARS` | 300 | Phase 2 — max running text length for merging |
| `TOPIC_SIGNAL_MIN_CUES` | 5 | Phase 4 — min cues before flushing at a boundary |
| `CHUNK_HARD_CAP_CUES` | 20 | Phase 4 — hard cap on cues per chunk |
| `SILENCE_GAP_BOUNDARY_SECONDS` | 8 | Phase 3 — silence gap triggers boundary |
| `OVERLAP_CUES` | 2 | Phase 4 — number of overlap cues between chunks |

## Phase 1: Parse the VTT File

### Step 1.1 — Read and clean the raw VTT

File: `src/ingestion/vtt_parser.py`

- Read `.vtt` file as text
- Strip the `WEBVTT` header line
- Remove all cue index numbers (standalone integers on their own line)
- Remove all timestamp lines (`HH:MM:SS.mmm --> HH:MM:SS.mmm`)
- Strip speaker labels from text (e.g. `PAUL:` → remove prefix up to and including `:`)
- Normalize line endings (`\r\n` → `\n`)
- Strip trailing whitespace from each line

### Step 1.2 — Parse into structured cue objects

Each cue becomes a dict:

```python
{
    "cue_id": int,           # sequential index
    "start_seconds": float,  # timestamp converted to total seconds
    "end_seconds": float,
    "start_timestamp": str,  # original "HH:MM:SS.mmm"
    "end_timestamp": str,
    "text": str              # cleaned text
}
```

- Convert timestamps to total seconds for gap arithmetic
- Store original timestamp strings for metadata
- Expected: ~2,357 cue objects for a 4.5hr recording

## Phase 2: Merge Micro-Cues into Sentences

### Step 2.1 — Merge consecutive short cues

File: `src/ingestion/cue_merger.py`

Many cues are just 1–2 words (e.g. `"PAUL: Perfect."`, `"PAUL: Alright."`).

Merge rules (applied sequentially through cue list):

1. If the gap between two consecutive cues is **> MERGE_GAP_THRESHOLD_SECONDS** → do NOT merge (break here)
2. If the running merged text would exceed **MERGE_MAX_CHARS** → do NOT merge (break here)
3. If the current cue ends with `.`, `?`, or `!` AND the next cue starts a new sentence (capital letter) → prefer to keep as a boundary (do NOT merge)
4. Otherwise → merge: append text, extend `end_seconds`/`end_timestamp` to the later cue

Output: fewer, fuller "sentence-level" cues, still with `start_seconds`/`end_seconds` timestamps.

## Phase 3: Detect Topic Boundaries

### Step 3.1 — Define topic-signal keyword patterns

File: `src/ingestion/topic_detector.py`

Trigger phrases (case-insensitive):

```python
TOPIC_SIGNAL_PHRASES = [
    "alright so", "alright,", "okay so", "so now", "now let",
    "next,", "next step", "the next", "let me show", "let me open",
    "let me explain", "let me just", "let me go", "moving on",
    "so what we", "so the idea", "so basically", "so here",
    "in summary", "to summarize", "so this is", "this particular",
    "notebook", "step ", "part ", "section",
]
```

### Step 3.2 — Score each cue for topic-shift likelihood

For each merged cue:

1. Check if its text contains any trigger phrase → if yes, `potential_boundary = True`
2. Check if it is the cue immediately after a silence gap > **SILENCE_GAP_BOUNDARY_SECONDS** → if yes, `potential_boundary = True`

Output: each cue gets a `potential_boundary: bool` field.

---

## Phase 4: Build Chunks

### Step 4.1 — Chunk creation rules

File: `src/ingestion/chunk_builder.py`

Maintain a `current_chunk_buffer` (list of cues). For each cue:

1. Add cue to buffer
2. If `potential_boundary == True` AND `len(buffer) >= TOPIC_SIGNAL_MIN_CUES` → flush buffer as a chunk, start new buffer
3. If `len(buffer) >= CHUNK_HARD_CAP_CUES` → flush buffer as a chunk regardless, start new buffer

After all cues processed → flush remaining buffer as the final chunk.

### Step 4.2 — Add overlap between chunks

After all chunks are created, go back and:

- Prepend the last **OVERLAP_CUES** cues of chunk N to the start of chunk N+1
- Mark overlap cues with `is_overlap: True` so they are not double-counted in display but ARE included in embedding

---

## Phase 5: Enrich Each Chunk with Metadata

### Step 5.1 — Compute per-chunk metadata

File: `src/ingestion/metadata_enricher.py`

Every chunk object must contain:

```python
{
    "chunk_id": int,              # sequential (0-indexed)
    "text": str,                  # full joined text of all cues in chunk
    "start_time": str,            # timestamp string of first cue (HH:MM:SS)
    "end_time": str,              # timestamp string of last cue (HH:MM:SS)
    "start_seconds": float,
    "end_seconds": float,
    "duration_seconds": float,
    "cue_count": int,
    "topic_hint": str,            # first sentence (up to first period or 120 chars)
    "chunk_type": str,            # "code_walkthrough" | "qa" | "explanation"
    "source_file": str,           # original .vtt filename
}
```

### Step 5.2 — Auto-tag chunk_type

Classify each chunk based on keyword presence in its text:

| Type | Trigger Keywords |
|---|---|
| `"code_walkthrough"` | `notebook`, `execute`, `cell`, `import`, `function`, `def `, `class `, `pip install` |
| `"qa"` | `question`, `anyone`, `let me know`, `in the chat`, `yes,`, `no,` |
| `"explanation"` | default if neither above matches |

---

## Phase 6: Output & Validation

### Step 6.1 — Serialize chunks

File: `src/ingestion/pipeline.py`

- Write all chunks to `data/chunks/chunks.json` — array of chunk objects
- Write `data/chunks/chunks_summary.json` with just: `chunk_id`, `start_time`, `end_time`, `topic_hint`, `chunk_type`, `cue_count`

### Step 6.2 — Validate output

Assertions:

1. Total chunks is between 80–200 (reasonable for a 4.5hr session)
2. No chunk has text under 100 characters (too small → merge logic failed)
3. No chunk has text over 3,000 characters (too large → cap logic failed)

Print summary:

```
Total chunks: 142
Avg chunk duration: 114.3s
Chunk type distribution:
  code_walkthrough: 38
  qa: 29
  explanation: 75
```

## Pipeline Orchestrator

File: `src/ingestion/pipeline.py`

```python
def run_vtt_chunking_pipeline(vtt_file_path: str, output_dir: str = "data/chunks") -> dict:
    """Run the full VTT chunking pipeline.

    Args:
        vtt_file_path: Path to the .vtt file
        output_dir: Directory for output JSON files

    Returns:
        dict with 'chunks', 'summary', and 'validation' keys
    """
```

Steps:

1. Parse VTT → `vtt_parser.parse_vtt(file_path)`
2. Merge cues → `cue_merger.merge_cues(cues)`
3. Detect boundaries → `topic_detector.detect_boundaries(merged_cues)`
4. Build chunks → `chunk_builder.build_chunks(cues_with_boundaries)`
5. Enrich metadata → `metadata_enricher.enrich(chunks, source_file)`
6. Validate & output → write JSON files + print summary

## Notes

- All constants are exposed via `config.py` for easy tuning without code changes
- The pipeline is file-based (no DB dependency at this stage)
- Overlap cues ensure embedding continuity across chunk boundaries
- `chunk_type` auto-tagging is keyword-based; can be upgraded to LLM-based classification later
