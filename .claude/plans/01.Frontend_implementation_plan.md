# Frontend Implementation Plan: Streamlit RAG Upload Interface

## Overview
Create a polished, multi-step Streamlit application for uploading class materials including VTT subtitle files, code files, and GitHub repository links. All uploaded content will be saved to a `docs/` folder for downstream processing.

---

## UI/UX Design Goals

1. **Clean, Professional Look**
   - Use Streamlit's wide layout mode for spacious design
   - Custom CSS for branded header, styled cards, and improved typography
   - Consistent color scheme (primary: blue/indigo accent)

2. **Step-by-Step Flow**
   - Progress indicator showing current step
   - Clear navigation (Continue/Back buttons)
   - Summary review before final submission

3. **Visual Feedback**
   - File upload progress indicators
   - Success/error toasts with icons
   - File type icons for uploaded items

---

## Step-by-Step Interface Flow

### Step 0: Landing Page
- App title and brief description
- "Get Started" button to begin
- Optional: Feature highlights (3 cards showing what the app does)

### Step 1: Upload VTT Subtitle Files
**Header:** "Upload Class Recording Subtitles"

**Elements:**
- File uploader accepting `.vtt` files (multiple allowed)
- Display list of uploaded files with:
  - File name
  - File size
  - Remove button per file
- Validation: Ensure at least one VTT file is uploaded before proceeding
- Preview section: Show first 500 characters of first uploaded file (collapsible)

**Actions:**
- "Continue to Code Files" button (enabled when ≥1 file uploaded)

### Step 2: Upload Code Files (Optional)
**Header:** "Do you have any code files discussed in the class?"

**Elements:**
- Yes/No toggle or button selection
- If "Yes":
  - File uploader accepting: `.py`, `.js`, `.ts`, `.jsx`, `.tsx`, `.java`, `.cpp`, `.c`, `.go`, `.rs`, `.rb`, `.php`, `.sql`, `.html`, `.css`, `.json`, `.yaml`, `.yml`, `.md`
  - Multiple files allowed
  - Display grid/list of uploaded code files with language icons
  - Remove button per file
- If "No": Skip to Step 3

**Actions:**
- "Continue to GitHub" button
- "Back" button to return to Step 1

### Step 3: GitHub Repository Links (Optional)
**Header:** "Do you have any GitHub repository links?"

**Elements:**
- Yes/No toggle or button selection
- If "Yes":
  - Dynamic input field for GitHub URLs
  - "Add Another" button to add more URL fields
  - Each URL field has:
    - Text input for GitHub URL (validation: must be github.com domain)
    - Remove button (if more than one field)
  - Display validation feedback (green checkmark for valid URLs, red X for invalid)
  - Optional: Branch selector (defaults to main/master)
  - Display summary: "X repositories will be fetched"
- If "No": Skip to review

**Actions:**
- "Review and Submit" button (enabled when all URLs are valid or empty)
- "Back" button to return to Step 2

### Step 4: Review and Submit
**Header:** "Review Your Uploads"

**Elements:**
- Summary cards showing:
  - VTT files: Count + list of filenames
  - Code files: Count + list of filenames (or "None uploaded")
  - GitHub repos: Count + list of repo names/URLs (or "None provided")
- Editable: "Edit" link next to each section to jump back to that step
- Large "Process and Save" button (primary action)

**On Submit:**
- Show progress spinner with text: "Saving files to docs/ folder..."
- Step indicators:
  1. "Creating docs/ folder..."
  2. "Saving VTT files..."
  3. "Saving code files..."
  4. "Fetching GitHub repositories..."
  5. "Organizing file structure..."
- On success: Show success message with checkmark icon, summary of saved files
- On error: Show error toast with details and "Retry" option

### Step 5: Success/Completion
**Header:** "Upload Complete!"

**Elements:**
- Large success checkmark animation
- Summary of what was saved:
  - Total files saved to `docs/` folder
  - Breakdown by type (VTT, code, GitHub)
- "Start New Upload" button (resets the form)
- "Go to Chat" button (future: links to RAG chat interface)
- Optional: "Open docs/ folder" button (reveals file location)

## Technical Implementation Details

### State Management
- Use `st.session_state` to track:
  - `current_step`: Current step in the flow (0-5)
  - `vtt_files`: List of uploaded VTT files
  - `code_files`: List of uploaded code files
  - `github_urls`: List of GitHub repository URLs
  - `has_code_files`: Boolean for code file step
  - `has_github_repos`: Boolean for GitHub step
  - `processing`: Boolean for submission state
  - `completed`: Boolean for completion state

### Custom Styling
- Inject custom CSS via `st.markdown()` with `unsafe_allow_html=True`:
  - Custom header with gradient background
  - Styled cards with shadows
  - Improved button styles
  - File upload area styling
  - Progress step indicator

### Helper Functions
- `save_uploaded_files()`: Saves uploaded files to docs/ folder
- `fetch_github_repo()`: Clones/fetches GitHub repository content
- `validate_github_url()`: Validates GitHub URL format
- `reset_form()`: Resets all session state to initial values

### Error Handling
- File upload errors (size limits, format validation)
- GitHub URL validation errors
- File system errors (permissions, disk space)
- Network errors for GitHub fetching
